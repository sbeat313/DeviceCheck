# DeviceCheck 系統說明

## 1. 系統目標
DeviceCheck 用來定期探測多台設備（以 UID 辨識），並在「確認異常（Dead）」狀態發生或解除時送出通知，避免因瞬間抖動造成過度告警。

---

## 2. 架構與元件
- **DeviceCheck（主服務）**
  - 提供設備查詢與別名維護 API。
  - 背景服務週期探測設備狀態。
- **DeviceCheck.Notifier（通知接收器）**
  - 接收主服務送出的狀態轉換通知。
  - 以日誌方式記錄通知內容。

---

## 3. 狀態模型
設備狀態包含：
- `Alive`：探測成功。
- `Busy`：設備忙碌（486）。
- `Dead`：確認異常。
- `Unknown`：狀態不明（例如 dead 次數尚未達門檻）。

### Dead 判定規則
1. 探測為 `Dead` 時，會累積連續 dead 次數。
2. 連續 dead 次數 **達到 `DeadConsecutiveThreshold`** 才會正式標記為 `Dead`。
3. 未達門檻前，狀態以 `Unknown` 呈現。

---

## 4. 通知規則
通知僅在「是否為 Dead」邊界變化時觸發：
- 非 `Dead` -> `Dead`：發送「確認異常」通知。
- `Dead` -> 非 `Dead`：發送「異常恢復」通知。
- `Alive` / `Busy` / `Unknown` 之間互轉：不通知。

> 通知內容中的非 Dead 狀態會統一呈現為 `Alive`，讓訊息語意集中在異常邊界。

---

## 5. 設定檔策略
- 業務設定集中在 `config.json`：
  - `DeviceCheck`（監控、探測、UID、別名）
  - `Notification`（通知開關與端點）
- `appsettings.json` 保留 ASP.NET Core / 日誌等通用設定。

---

## 6. API 與可觀測性
- 主服務已啟用 Swagger UI（`/swagger`）。
- 可透過 API：
  - 心跳回報
  - 設備狀態查詢
  - UID 別名更新（同步回寫 `config.json`）
