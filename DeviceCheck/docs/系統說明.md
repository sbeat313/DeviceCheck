# DeviceCheck 系統說明

## 1. 系統目標
DeviceCheck 用來定期探測多台設備（以 UID 辨識），並在「確認異常（Dead）」狀態發生或解除時送出通知，避免因瞬間抖動造成過度告警。

---

## 2. 架構與元件
- **DeviceCheck（主服務）**
  - 提供設備查詢與別名維護 API。
  - 背景服務週期探測設備狀態。
- **DeviceCheck.Notifier（通知接收器）**
  - 接收主服務送出的狀態轉換通知。
  - 以日誌方式記錄通知內容。

---

## 3. 狀態模型
設備狀態包含：
- `Alive`：探測成功。
- `Busy`：設備忙碌（486）。
- `Dead`：確認異常。
- `Unknown`：狀態不明（例如 dead 次數尚未達門檻）。

### Dead 判定規則
1. 探測為 `Dead` 時，會累積連續 dead 次數。
2. 連續 dead 次數 **達到 `DeadConsecutiveThreshold`** 才會正式標記為 `Dead`。
3. 未達門檻前，狀態以 `Unknown` 呈現。

---

## 4. 檢測流程（含 Dead/Alive 判定與 Retry）
每台設備到期後會執行探測（`/Ctrl/{uid}/RadioCheck`），狀態轉換如下：
- HTTP `200`：判定為 `Alive`。
- HTTP `486`：判定為 `Busy`。
- HTTP `503`、其他非 200/486、timeout、例外：判定為 `Dead`（探測結果層級）。

### 目前的 Dead/Alive 機制
1. 探測結果為 `Alive` 時，設備狀態更新為 `Alive`，並清空連續 dead 次數。
2. 探測結果為 `Busy` 時，設備狀態更新為 `Busy`，並清空連續 dead 次數。
3. 探測結果為 `Dead` 時，會累積 `ConsecutiveDeadCount`：
   - 未達 `DeadConsecutiveThreshold`：狀態為 `Unknown`。
   - 達門檻後：狀態才會正式進入 `Dead`。

### Retry 機制（新增）
當單次探測結果為 `Dead` 或 `Busy`，會在**同一輪檢測內**進行重試：
- `ProbeRetryCount`：重試次數（不含第一次呼叫）。
- `ProbeRetryDelaySeconds`：每次重試前等待秒數；若最終結果為 `Busy`，也用此值安排下次檢測。

範例：`ProbeRetryCount = 2` 時，單輪最多會呼叫 3 次（首次 + 2 次重試）。
若重試途中取得 `Alive`，會立即停止重試並採用該結果；若仍為 `Dead/Busy`，則採用最後一次結果進入後續狀態判定與排程。

> 已以 `ProbeRetryDelaySeconds` 統一取代舊的 `BusyRetryDelaySeconds`。

---

## 5. 通知規則
通知僅在「是否為 Dead」邊界變化時觸發：
- 非 `Dead` -> `Dead`：發送「確認異常」通知。
- `Dead` -> 非 `Dead`：發送「異常恢復」通知。
- `Alive` / `Busy` / `Unknown` 之間互轉：不通知。

> 通知內容中的非 Dead 狀態會統一呈現為 `Alive`，讓訊息語意集中在異常邊界。

---

## 6. 設定檔策略
- 業務設定集中在 `config.json`：
  - `DeviceCheck`（監控、探測、UID、別名）
  - `Notification`（通知開關與端點）
- `appsettings.json` 保留 ASP.NET Core / 日誌等通用設定。

---

## 7. API 與可觀測性
- 主服務已啟用 Swagger UI（`/swagger`）。
- 可透過 API：
  - 心跳回報
  - 設備狀態查詢
  - UID 別名更新（同步回寫 `config.json`）
